02

Nav footprint

Two elements. Brand-mark slot (left, B3 rip-mark) and one route in (right, "Sign in"). No nav menu, no marketing categories, no language switcher, no logo wordmark adjacent to the mark. Landing is identity-first; a marketing nav would soften the entry.

Layout

position: fixed at the page root. Container has pointer-events: none; children opt back in — so the empty middle column doesn't intercept clicks on whatever the rip-mark is doing under it.

The route uses the pill chrome from the chrome lib's vocabulary: hairline border, slime-soft tinted backdrop, blur. Snap easing on hover transitions (chrome scale, not landing-scale).

Why one route, not two

"Sign in" covers both new and returning. A "Sign up" button would split the eye and weaken the rip-mark's gravity. Routing to /signin handles new-user upgrade in-flow. Reversible if user research disagrees.

03

Hero

Three-row grid: breathing room, rip-mark band, copy stack. Bottom of the hero hosts a hairline + mono "Scroll" cue — no bouncing chevron, no animated arrow. Identity, not affordance theatre.

The void decision

Hero opens on void, not earth. Three reasons: A1's "make it yours." vignette already canonized void as Koy's hero substrate; rip-from-void grammar inherits literally from the project name; an earth+cosmos comp borrows recognizable composition from the DC-8 Sony reference, which Absolute Rule #6 fires hardest on at landing scale. Reversible if Khalid calls it back to earth.

Slots, not content

kl-hero__rip-slot is a square canvas, 320px / clamped to 60vmin. B3 fills with the rendered rip-mark (video / GIF / SVG-animated). NO live shader.

kl-hero__copy-slot reserves 720ch-max width with text-wrap: balance. The phrase token is --t-4xl (72px desktop, 48px mobile). B2 fills the final string; "make it yours." is shown here as voice-direction stand-in.

Why a "Scroll" cue at all

Landing is a multi-section scroll program. Visitors who don't scroll never see B5's templates and never reach the bridge. The cue is a single hairline + 10px mono label — no animation, no bounce. The scroll progress bar at the very top is the parallel wayfinding signal.

04

Scroll transition

Two pieces, both pure CSS. A slime-tinted hairline that fills left-to-right with document scroll, and a per-hotspot reveal-on-enter that translates each card up into final position as it crosses the viewport. Translate-only, no fade. Both wrapped in @supports (animation-timeline: …) so non-supporting UAs render the layout statically — no broken-looking page.

Why no JS scroll listeners

Constitution Section 5 + Decision #2: sandbox-safe means pure CSS. animation-timeline: scroll(root) ties the progress bar to document scroll without listeners; animation-timeline: view() ties each hotspot reveal to its own intersection with the viewport. Browser does the math.

NO fade

Hotspot reveal is translate-only. The cards enter at final opacity, rising 48px into place across the entry range entry 0% cover 30%. Scrim opacity ramps are the only sanctioned opacity transitions, and B1 has none — the progress bar fills via transform: scaleX, not opacity.

Reduced motion

prefers-reduced-motion kills both: progress bar pins to scaleX(0); hotspots cancel the rise animation. Layout still works; the page just doesn't kinetic.

kl-scroll-progress — 2px hairline, slime tint, fixed-top, animation-timeline: scroll(root). Scroll the doc to watch it fill.
kl-hotspot reveal — animation-timeline: view(), range entry 0% → cover 30%, translateY(48px → 0). Linear timing function so the motion feels scroll-controlled, not scroll-triggered.
05

Slime band slot

The inter-section void between hero and hotspots. B4 fills with the slime takeover asset — rendered, shipped flat, NOT a live shader. B1 reserves the band so the page rhythm reads correctly without B4 shipped.

Why a dedicated band, not a continuous bleed

Slime is the project's mascot moment. Burying it as background texture under hotspots dissolves its presence. Giving it its own band (~42vh / 280px floor) lets the takeover feel like an event that happens between sections — slime spreads ACROSS the void, then hotspots arrive. Authored chrome, not decoration.

Asset coordination — OPEN ITEM

B4 deliverable spec hasn't been written yet. Open: who renders the slime asset? Khalid (founder direction), external tool (paper.design or similar generator), or GPT (ad lane only)? B1 reserves the band; B4 closes the asset-coordination question.

06

Hotspots

Six slots. 3×2 desktop constellation; 2×3 phone; 1×6 ultra-narrow. Each hotspot is a doorway to a template (DC-9, founder-confirmed). B1 ships the grid + per-card scaffold; B5 fills the breathing / preview / slime-acknowledgement / route.

Constellation, not bento

The grid is a 12-col × auto-row scaffold; each card has its own (col, row, span). No two cards share a row baseline or a column edge. A symmetric 3×2 would hand the territory back to productivity-tool framing (Rule #4) — the asymmetry is the point. Unsupervised maximalism reaches into chrome here through composed irregularity.

Why 6

Six cards give enough mass for a star map without crowding. Four reads as a quad / decision matrix; nine reads as a feature grid. Six in this constellation lets the eye compose left→right in descending waves — not by row-major scan.

Per-card anatomy

  • head — mono number (01–06) · taxonomy tag (Purpose / Vibe / Commerce)
  • preview — striped placeholder; B5 fills with template mini-preview
  • foot — display label · mono sub-line

Card sizes vary by grid span (3–5 cols, 3–5 rows of a 64px row track). Background is a low-chroma slime-tinted oklch with backdrop-filter: blur(2px) so the void halo behind the grid bleeds through faintly.

Labels are stand-ins

"the page / the zine / the shop / the link / the manifesto / the pre-order" are placeholder labels chosen to demonstrate the voice direction (definite article + noun = identity-first, not productivity-tool-first). Phase C names them for real.

07

Bridge · hairline-reveal

A vertical hairline with transparent endcaps, a mono cue, and a single underlined CTA. The doorway from landing into the template gallery (Phase C). B5 wires the routing — hotspot click goes to template-detail; bridge CTA goes to gallery index.

Mechanism (one line)

Static <span>, 1px × 96px, background: linear-gradient(to bottom, transparent, var(--accent), transparent). No animation. No scroll binding. No opacity transition. The transparent endcaps ARE the "reveal" — the line appears to fade in and out of the void because the gradient endcaps match the void substrate.

Why a hairline, not a button

A primary-coloured button at the foot of landing would compete with the rip-mark for gravity. The bridge is a soft handoff — the user has scrolled past hero, slime, and the constellation; the bridge is the polite "and if none of those landed, here's all of them." A button shouts. A hairline + underlined link concedes.

08

Mobile reshape

Landing reshapes at <768px. NOT a desktop-trimmed view. The constellation can't survive narrow viewport, so it collapses to a single-column vertical scroll-reveal of cards — top-down by feature priority. Hero scales down (72 → 48 type, 320 → 220 rip-slot). Slime band compresses but stays present.
