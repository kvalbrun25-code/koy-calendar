01

Architecture

A 48px toolbar + tri-column body. The middle column is canvas (untouched). Both side columns are chrome — Blocks left, Inspector right. The canvas zone keeps the A1 vignette's dashed "not chrome's territory" marker for clarity in spec docs only.

DC-4 (Bloomberg half) · § PROTECTION 5 Dense where it should be · fast · the chrome is a frame, not a stage. Grid is auto 1fr rows, 264px / 1fr / 320px columns. The toolbar is 48px because anything taller starts to feel like a SaaS dashboard; anything shorter loses room for the state-comm surfaces the Save button needs to host. The Inspector is slightly wider than Blocks (320 vs 264) because per-block fields and the future Style/Solids/Gradients controls (DC-4) need more horizontal real estate than a list of draggable block types. The canvas zone sits on --bg-deep so the chrome (--bg-chrome) reads as a frame around it — chrome's slightly lighter surface is the visual signal of "this is the frame, not the work."

02

Editor toolbar

Five regions across one row: brand · mode tabs · page meta · actions · account. Density tuned to fit on a 13" laptop without crowding; structure stays semantic so screen-readers and keyboard nav read the same chrome.

DC-4 · § PROTECTION 5 Trustworthy, fast, never in the way. Mode tabs use a contained segmented control instead of underlines — segmented reads correctly on dark chrome where underlines disappear, and the slime accent on the selected tab is the only place the brand color enters the toolbar baseline. The page meta strip is a single line because density wins over hierarchy here: title · status (saved/draft/live) · version. The title is editable in place (click → text caret). The live status pulses (only when actually live) — the only ambient motion in chrome at rest; everything else is event-driven. The account chip on the right is a pill-radius because it sits in the densest part of the bar and the pill softens that one corner without rounding the rest of the chrome.

03

Save · state machine

The state-comm primitives live inside the Save button, not floating beside it. Idle → Saving → Saved → Idle, or Idle → Saving → Failed. The button itself is the truth surface for its own action — Invariant I5 made visible at the smallest possible scale.

STATE-COMM § 1–3 · INVARIANT I5 "No lies, no silence" · the action carries its own truth. A floating toast for "Saved" is two problems: it disconnects feedback from the action, and it competes for the same screen position as session-class errors. Wiring the success directly into the button removes both. Saving: the button stays in place (UI does not lock — the user can keep editing other surfaces), shows a spinner, sets cursor: progress; the label changes to "Saving…". Saved: the button takes the success tint (brand-dimmed slime) with a check, auto-collapses to idle after ~1.5s. Failed: the button takes the warn tint, label changes to "Couldn't save", and a small inline panel beneath the button reveals the technical line and a Retry CTA. The error stays sticky — it does not auto-dismiss — because that's a real recoverable error per state-comm primitive #3. Non-recoverable errors (session expired) escalate to the top toast anchor — not this button.

All four states, frozen interactive cycle below

idle
Default. Border-only chrome surface. No icon. Clicking starts the cycle.
saving
Spinner active. cursor: progress. Button disabled; rest of UI not locked. Pulse-shaped.
saved
Brand-dimmed slime tint. Holds ~1.5s, then collapses to idle. The page meta strip's "SAVED · just now" updates on the same beat.
failed
Couldn't save — try again in a sec. 409 · PAGE CONFLICT (someone else edited)
Warn-amber tint. Sticky inline reveal with plain-English + technical line behind it. Per state-comm rules — never silent.

Interactive cycle click → idle · saving · saved · idle. Shift-click → idle · saving · failed.

04

Side panels

Sticky 40px header · scrolling body · collapsible sections. Blocks-left has search and grouped categories (Content / Live / Advanced). Inspector-right shows the empty state when nothing is selected — never a blank pane.

DC-4 (structural skeleton) · § PROTECTION 3 Dignified empty states · never punishing. Panel headers stick on scroll because in long block lists or inspector forms, losing the panel title costs more than the 40px it claims. Section heads use a chevron rotation (snap, 160ms) — no fancy accordion collapse animation; instant disclosure preserves Bloomberg pacing. Block items use a 24px square glyph (mono-character placeholder for production icons) — placeholder is intentional and called out so Dev A doesn't ship rendered emoji or stylized icons unless an icon set lands. The Inspector empty state is the most-design-relevant surface here: a creator with nothing selected sees a dignified hint, not "select something" scolding. The hint includes a keyboard hint so the empty state teaches as well as waits — chrome that's a little smarter than it needs to be.

05

Modal · destructive confirm only

Modals are reserved for the one thing that genuinely needs to interrupt: destructive confirms. Everything informational lives inline or in a toast (§ 06). Snap entrance; backdrop-blur scrim; Esc + click-outside + explicit Cancel all dismiss; focus traps in, returns to invoker.

STATE-COMM § 5 · § PROTECTION 5 (Modern UX Discipline) Interruption costs trust · spend it only on real loss. The modal's entrance uses snap (160ms, ease-out hard) — it's a UI landing, not an ambient settle, so it arrives definitively. The scrim uses backdrop-filter: blur + a void-tinted overlay so the underlying chrome doesn't disappear — the modal sits over a softened reality, not in darkness. Focus management is non-negotiable: focus moves to the safest action (typically Cancel) on open, traps within the dialog, returns to the invoker on close. Esc dismisses (with Cancel semantics, not destructive). Click-outside dismisses. The destructive action stays as a plain --btn--danger — no shake, no big red — because the dignity of the affordance matters at the moment a user is about to lose something. The InlineConfirm pattern from A1 covers smaller destructive moments inline; full modal is reserved for irreversibles where the consequence merits the interruption.

06

Toast & state-comm anchors

Three canonical positions for state communication — inline-at-action, toast-top, modal-center. Production code references the named anchor, not pixel coordinates. A3 will spec the full primitives; A2 locks the chrome positions they live inside so A3 can't redecide layout.

STATE-COMM § ROLE RULES Truth at the action · never two channels for the same event. The toast anchor sits centered, just below the toolbar. Anchoring centered (not top-right) is deliberate — a session-class error needs to be acknowledged, not dismissed reflexively from the corner. Glide entry (ambient surface), not snap (which is reserved for the foreground "this landed" feeling). Toasts never auto-dismiss for non-recoverable errors — Sign-back-in and Reconnect carry CTAs the user must act on. Background events ("Auto-saved", "Refresh fired") may auto-dismiss after 4s with a success tone. The table below names every state-comm event and where its chrome anchor lives. A3 inherits this without redeciding.

07

Navigation frame

Two affordances: the account menu (right end of toolbar) and the page switcher (clicking the page title). Landing nav is Phase B — A2 only covers in-app movement.

DC-1/2/3 soul · § PROTECTION 6 (policy stability) Movement that respects how possessive a creator gets about their pages. The account menu is dropdown-shaped, not a slide-in drawer — drawer animations on a non-mobile surface tax for nothing. The page switcher mounts from the title (not a separate "switch page" button) because the title is the strongest signifier of "the page I'm in"; clicking it to see siblings reads as a creator gesture, not a navigation widget. Status badges (LIVE / DRAFT) in the switcher tell the truth at-a-glance — no clicking through to find out which version visitors actually see, which was the wound Sprint 2's B12 was about. Sign-out lives in its own group at the bottom of the menu, with the destructive tint reserved for it.

Both menus live · click the toolbar's account chip or page title esc / click-outside dismisses

08

Mobile chrome  < 768PX

Single-column. Drag DISABLED per the bible — same blocks, different rendering. The toolbar condenses to brand + title + Publish. Mode tabs descend to a bottom bar. Blocks and Inspector become bottom action sheets.

§ TECHNICAL FACTS · § PROTECTION 5 A phone is a real medium, not a smaller desktop. The mobile breakpoint forces a real reshape, not a CSS-truncated desktop. The toolbar's mode tabs migrate down because a thumb reaches up less comfortably than across; the mode bar at the bottom keeps the most-frequent gesture (Edit ↔ Preview) one tap away. Inspector and Blocks promote to action sheets sliding up from the bottom — glide easing, snapping into a max-80vh height. The grip handle is non-functional decoration in v1 but reserves real-estate for swipe-to-dismiss in the sandbox sprint downstream. The "live" status pulses still — it's the only ambient motion that survives the mobile reshape, because a creator on mobile especially needs to know what visitors are seeing.
